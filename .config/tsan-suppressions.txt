# ThreadSanitizer suppressions
# ─────────────────────────────────────────────────────────────────────────────
# Loaded by the `tsan` CI job (see .github/workflows/ci.yml) via
#   TSAN_OPTIONS=...:suppressions=<this file>
# halt_on_error stays on, so any race or deadlock NOT matched here still fails
# CI. This file matches exactly one frame, on the B+ tree structural-writer
# write-latch path, and nothing else.
#
# What is suppressed
# ------------------
# `deadlock:` (lock-order-inversion) reports whose cycle runs through
# entropy::WriteSet::relatch_entry -- the single helper through which every
# structural writer (insert / remove / split / merge) takes a Page write latch
# while crabbing down the tree (b_plus_tree.cpp:193, reached via
# WriteSet::acquire / track / adopt). It is a file-local writer-only helper: it
# is never on any reader path (readers use Page::rlatch, a different frame).
#
# Why these reports are false positives
# -------------------------------------
# 1. Frame-address reuse. ThreadSanitizer identifies each per-page reader/writer
#    latch (Page::latch_, a std::shared_mutex) by its ADDRESS. The buffer pool
#    owns a fixed array of frames and rebinds a frame's Page object to different
#    logical pages over an operation's lifetime (evict + reallocate). The latch
#    lives in the Page and survives that rebind (Page::reset never touches it,
#    and a latched page stays pinned so it cannot be evicted under its holder),
#    so over time one frame address stands in for many DISTINCT logical page
#    latches. TSan cannot see that distinction: it folds the acquisition orders
#    of unrelated logical latches that merely shared a frame address into one
#    lock-order graph and reports a cycle. Several of the flagged tests are
#    SINGLE-THREADED (the report says "in main thread"). A lone thread cannot
#    deadlock against itself, which is direct proof the cycle is an artifact of
#    address reuse, not a real ordering hazard.
#
# 2. Writers are serialized, so writer-vs-writer cannot deadlock. Every frame in
#    every one of these reports is a Page write latch taken via relatch_entry.
#    insert() and remove() each run under BPlusTree::write_mutex_
#    (std::lock_guard), so at most one structural writer holds page write latches
#    at any instant. A deadlock needs two parties that hold-and-wait on each
#    other. With a single serialized writer there is no second writer to close a
#    real cycle. Every writer-only inversion here is therefore benign by
#    construction, which is why matching this one writer frame cannot hide a real
#    writer-vs-writer deadlock: none can exist.
#
# 3. The reader-vs-writer inversion is gone. Round-1 review caught a real
#    reader-vs-writer deadlock (leaf-chain scans vs delete/insert churn). It was
#    fixed structurally, not papered over: parent-pointer maintenance
#    (set_parent / reparent_children) is now latch-free -- parent_page_id is
#    written and read only by serialized structural writers, readers never touch
#    it -- and a failed borrow-from-right now releases the right sibling before
#    turning left. Confirming the fix holds: NONE of the suppressed reports
#    contain a single reader frame (Page::rlatch, descend_to_leaf_read,
#    range_scan). They are 100% writer-latch frames.
#
# Why this does not weaken detection
# ----------------------------------
# TSan matches a `deadlock:` suppression if any frame of the cycle matches, so
# this entry is deliberately confined to the writer-only relatch_entry frame,
# which is present in a report only when a structural writer participates in the
# cycle. A genuine reader-involved inversion additionally exercises the reader
# latch path (Page::rlatch via descend_to_leaf_read / range_scan) which this
# file does not mention. Reintroducing a reader-side latch inversion still trips
# TSan (verified). Data-race reports are a different report type entirely and are
# never matched by a `deadlock:` entry, so real races continue to fail CI.
deadlock:entropy::WriteSet::relatch_entry
